# Security

Vite's defaults are mostly safe. The dangerous patches all involve env vars and source maps.

## `VITE_` Prefix Is NOT a Security Boundary

Any `VITE_`-prefixed var is **statically inlined into the client bundle as plain text**. Minification, base64 encoding, removing source maps — none of it hides the value.

```ts
// VITE_API_KEY=sk_live_abc123 in .env

// In source
fetch('/api', { headers: { 'X-Key': import.meta.env.VITE_API_KEY } })

// In built JS — fully visible
fetch("/api",{headers:{"X-Key":"sk_live_abc123"}})
```

Anyone with the JS file can extract any `VITE_` var.

**Rule**: only public values (API URLs, feature flags, public keys, analytics IDs) go in `VITE_` vars. Secrets stay server-side behind an API or serverless function.

## The `loadEnv('')` Trap

```ts
// CATASTROPHIC — loads ALL env vars including server secrets, makes them
// available for `define`-inlining into client code.
const env = loadEnv(mode, process.cwd(), '')


// SAFE — explicit prefix list
const env = loadEnv(mode, process.cwd(), ['VITE_', 'APP_'])
```

Never pass `''` as the third argument. Never set `envPrefix: ''` in the config. Never add server-only env vars to `define`.

## Don't Inline Secrets via `define`

```ts
// BAD — bakes secret into client bundle
define: {
  __ADMIN_TOKEN__: JSON.stringify(process.env.ADMIN_TOKEN),
}
```

Same logic as `VITE_` vars: anything injected via `define` ships in the bundle. Read by attacker, used to impersonate.

## Source Maps in Production

Production source maps leak your original source code:

```ts
build: {
  sourcemap: false,    // default — keep this for prod
}
```

If you upload to an error tracker (Sentry, Bugsnag), use `'hidden'`:

```ts
build: {
  sourcemap: 'hidden',   // generates the file but doesn't reference it from JS
}
```

…then upload `dist/assets/*.map` to your tracker and **delete locally** before deploying. The browser won't fetch the map (no `//# sourceMappingURL` comment), so users can't see your source.

## Restrict Dev Server Access

```ts
server: {
  host: 'localhost',     // default — bind to localhost only
  // host: true,         // 0.0.0.0 — only when needed (Docker, mobile testing)
}
```

`host: true` exposes your dev server on every network interface. Anyone on the same network can connect. Use only when needed and assume the dev server is reachable while running.

## File System Allow

By default, Vite restricts file serving to project root. Don't widen unnecessarily:

```ts
server: {
  fs: {
    allow: ['..'],                           // monorepo root — fine
    deny: ['.env', '.env.*', '*.{pem,crt}'], // explicit blocks
  },
}
```

A misconfigured `fs.allow` can expose `.env`, certs, anything Vite can read. Add `deny` patterns explicitly.

## Strict CORS in Dev

```ts
server: {
  cors: { origin: 'https://your-other-app.dev', credentials: true },
}
```

`cors: true` (any origin) is fine for solo development; don't deploy. `vite preview` is **not** a production server — it carries the same `cors: true` default if set.

## CSP Compatibility

If you ship Content Security Policy headers:

- Vite's dev mode injects inline scripts for HMR — needs `'unsafe-inline'` in dev.
- Production builds can be CSP-strict. Use `'sha256-...'` hashes or a nonce, generated by your server.
- Avoid `eval` and `Function('...')` in your code; ESLint plugin `no-eval` catches.

## `.gitignore` Checklist

```gitignore
# Local env / secrets
.env.local
.env.*.local

# Build output
dist/

# Pre-bundle cache (shouldn't have secrets but is gitignore'd anyway)
node_modules/.vite

# Profile dumps
*.cpuprofile

# Optional — IDE / OS
.DS_Store
.vscode/
.idea/
```

Verify with `git check-ignore`:

```bash
git check-ignore -v .env.local         # should print the .gitignore rule
git check-ignore -v .env               # should NOT match (committed template is OK)
```

## XSS in Dev Middleware

If you write custom dev middleware, sanitize:

```ts
// BAD — reflects user input into HTML
configureServer(server) {
  server.middlewares.use('/echo', (req, res) => {
    const q = new URL(req.url, 'http://x').searchParams.get('q') ?? ''
    res.end(`<p>${q}</p>`)   // XSS
  })
}


// GOOD — escape, or use JSON only
res.setHeader('content-type', 'application/json')
res.end(JSON.stringify({ q }))
```

Dev middleware bugs are dev-only, but they can leak local info to other origins via CSRF if dev server is exposed.

## `eval` in Dependencies

A dep that uses `eval` or `new Function()` may not work under strict CSP and may break tree-shaking. Audit suspicious dependencies; reach for alternatives.

## Subresource Integrity (SRI) for CDN Scripts

If you load scripts from external CDNs in `index.html`:

```html
<script
  src="https://cdn.example.com/lib.js"
  integrity="sha384-..."
  crossorigin="anonymous"
></script>
```

Compute the hash; browsers refuse to execute the script if it doesn't match. Protects against CDN compromise.

## Update Dependencies

```bash
npm audit
npm outdated
```

Or use Dependabot / Renovate for automated PRs. Most "Vite security incidents" are downstream dep CVEs, not Vite itself.

## Pre-Launch Checklist

- [ ] No `loadEnv` with `''` prefix.
- [ ] No secrets in `VITE_*` vars.
- [ ] `build.sourcemap: false` (or `'hidden'` if you upload separately).
- [ ] `define: {}` contains no secrets.
- [ ] `.env.local` and `.env.*.local` gitignored.
- [ ] No `host: true` slipped into prod build.
- [ ] `vite preview` confirms the bundle starts and assets are served correctly.
- [ ] CSP / security headers configured at the static host (not in Vite config).
- [ ] `npm audit` clean (or known issues documented).
